# Аудит финансов, кода и продакшена — 2026-07-09

Цель: проверить CRM `racpechatca` как систему, которую дальше можно безопасно
масштабировать: деньги, бизнес-логику, кодовую архитектуру, продакшен и
операционные риски.

## Короткий вывод

Система в рабочем состоянии: продакшен поднят, БД отвечает, бэкапы пишутся,
финансовые итоги заказов сходятся с сохранёнными позициями, отрицательных сумм
и переплат зарплаты в базе не найдено.

Главные улучшения этого захода:

- добавлен публичный `GET /health` с проверкой PostgreSQL;
- добавлены Docker healthcheck-и для `postgres`, `backend`, `frontend`;
- `PrismaService` вынесен в единый глобальный `PrismaModule`, чтобы не плодить
  лишние Prisma-клиенты и подключения к БД;
- backend-валидация усилена для денег, количеств, пагинации и начального
  статуса заказа;
- nginx проксирует `/health` в backend;
- оба Compose-файла синхронизированы по Telegram env, Telegram `extra_hosts` и
  healthcheck-логике.

## Финансовый аудит

Проверено на production-БД:

```text
totalOrder vs sum(pricePosition) + deliveryCost: расхождений нет
отрицательные суммы/количества: нет
переплаты по зарплатным начислениям: нет
активные дубли начислений по одному заказу: нет
несовпадение pending/settled зарплаты с текущей суммой заказа: нет
SENT/PAID без sentAt: нет
отрицательный склад: нет
складовые движения у не-SENT заказов: нет
```

Найдены старые позиции, где `pricePosition` не равен текущей формуле позиции.
Итоги заказов при этом сходятся с сохранёнными `pricePosition`. Это похоже на
legacy/ручные договорные суммы, поэтому данные не переписывались автоматически.

Риск закрыт для новых данных усиленной backend-валидацией:

- цена и доставка не могут быть отрицательными;
- количество не может быть меньше `1`;
- суммы принимаются как целые числа;
- при создании заказа допустимы только стартовые статусы `LEAD` и `NEW`.

## Кодовый аудит

Сильные стороны:

- деньги считаются на backend, frontend не является источником истины;
- мутации позиций и пересчёт total идут через транзакции;
- есть блокировка редактирования финансов после частичной/полной выплаты;
- зарплатные выплаты закрывают начисления FIFO или выбранным списком;
- склад списывается идемпотентно при переходе в `SENT`;
- review reminder теперь учитывает `PHOTO/TSHIRT` и `SENT/PAID`.

Улучшено:

- один `PrismaModule` вместо локальных Prisma-провайдеров в каждом feature-модуле;
- health endpoint отделён от защищённых CRM API и пригоден для мониторинга;
- DTO контракты стали жёстче, чтобы API нельзя было использовать для ввода
  финансово невозможных значений.

Оставшиеся технические риски:

- frontend-типы всё ещё вручную повторяют backend/Prisma enum-ы;
- `getOrderStats` читает все подходящие заказы в память, при росте до десятков
  тысяч заказов лучше заменить на SQL/Prisma агрегации;
- отчёты P&L считают всё на лету; при росте стоит добавить materialized summary
  или периодические агрегаты;
- нет e2e-теста на полный сценарий заказа: создание -> отправка -> зарплата ->
  оплата -> отчёт.

## Продакшен-аудит

Текущее состояние сервера:

```text
disk: 50G total, ~39% used
memory: 1.6GiB total, без swap
containers: postgres/backend/frontend up
backend memory: ~113MiB
postgres memory: ~28MiB
frontend memory: ~4MiB
backups: локальные gzip-дампы пишутся в /opt/raspechatka/backups
```

Улучшено:

- Compose теперь ждёт healthy PostgreSQL перед backend;
- frontend ждёт healthy backend;
- backend healthcheck проверяет `/health`, а `/health` проверяет БД;
- frontend healthcheck проверяет nginx по `127.0.0.1`, чтобы не зависеть от
  IPv6-резолва `localhost` внутри контейнера;
- production-like и prebuilt Compose приведены к одинаковой логике Telegram env
  и Telegram IPv4 pinning.

Оставшиеся production-риски:

- swap отсутствует; при всплесках памяти лучше добавить небольшой swapfile;
- бэкапы лежат на том же сервере, нужен off-site backup;
- публичный доступ идёт по HTTP, для нормальной эксплуатации нужен HTTPS и домен;
- нет внешнего мониторинга доступности `/health`;
- нет автоматизированного deploy script с единым протоколом
  `backup -> pull -> build -> up -> health`.

## Следующие улучшения

1. Вынести frontend/backend типы API в общий контракт или генерацию.
2. Добавить `/admin/audit` или CLI-команду финансового аудита, которая регулярно
   проверяет расхождения total/зарплаты/склада.
3. Перевести статистику главной на агрегирующие SQL-запросы.
4. Добавить off-site backup: S3/облако/второй сервер.
5. Добавить swap и внешний мониторинг `/health`.
6. Добавить e2e-тест полного денежного жизненного цикла заказа.
