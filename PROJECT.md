# CRM «Распечатка» — как всё устроено

Актуально на 24 июля 2026. Документ описывает систему такой, какая она есть
сейчас, включая слабые места.

---

## 1. Что это

Внутренняя CRM магазина печати. Два продукта:

- **фотопечать** — делаем сами, работу выполняет исполнитель;
- **печать на футболках** — производство целиком у партнёра, мы принимаем
  заказ, передаём задание и рассчитываемся.

Система ведёт заказ от обращения клиента до оплаты, считает деньги (чек,
вознаграждение партнёра, зарплату исполнителя), напоминает о делах в Telegram
и хранит отчётность.

Основной канал продаж — Авито. Аккаунт «Распечатка», рейтинг 4.9, 221 отзыв.

## 3. Продукты и статусы

Потоки разные, потому что процессы разные.

**Фотопечать**

```
Обратился → Новый → Создана папка → В обработке → Готов → Отправлен → Оплачен
```

**Футболки** (короче: производство ведёт партнёр)

```
Новый → Отправлен (передали партнёру) → В работе (партнёр печатает)
      → Готов → Оплачен (рассчитались с партнёром)
```

Статусы футболок подтягиваются от партнёра автоматически: раз в 30 секунд
система опрашивает его систему и двигает заказ вперёд. Назад не двигает
никогда — оплаченный заказ не может «разоплатиться» из-за чужой ошибки.

**Защита:** перевести фотозаказ в «Отправлен» без назначенного исполнителя
нельзя. Иначе зарплата не начислится никому и это всплывёт через месяц.

## 4. Деньги

Самая ответственная часть системы. Все суммы считаются на сервере — то, что
прислал браузер, в расчёт не идёт.

### Чек клиента

```
Итог = позиции + доставка + разработка дизайна
```

Позиция считается по-разному:

- **обычная**: цена × количество;
- **договорная**: сумма равна введённой цене, количество не умножается
  (так оформляется фотопечать — клиент торгуется за всю пачку).

### Расчёт с партнёром (футболки)

Партнёр получает свои материалы обратно плюс долю от прибыли с печати:

```
материалы     = (термоплёнка + заготовка) × количество
прибыль       = цена позиции − разработка дизайна − материалы
доля партнёра = прибыль × 30 %
вознаграждение = доля партнёра + материалы
```

Пример: футболка 1500 ₽, заготовка 260, плёнка 70, дизайна нет.

```
материалы = 330
прибыль   = 1500 − 330 = 1170
доля      = 1170 × 0,3 = 351
партнёру  = 351 + 330 = 681 ₽
```

Заготовку по умолчанию даёт партнёр. Если клиент принёс свою вещь —
стоимость заготовки из расчёта уходит.

**Разработка дизайна партнёру не отдаётся** — это заработок магазина.
Поэтому она отдельная строка чека, а не надбавка к цене позиции. Если
подмешать её в цену, партнёр получит долю с чужих денег.

### Зарплата

- **Исполнитель** получает процент от базы заказа, где база = чек минус
  доставка минус дизайн.
- **Менеджер по оформлению** — свою ставку за оформленный заказ плюс премию
  процентом от суммы «разработка дизайна».

Начисление возникает при переходе в **«Отправлен»** и гасится при выплате.
Уход из «Отправлен» снимает незакрытое начисление; если по нему уже платили —
система не даст откатить статус молча.

Исполнитель видит свой заработанный остаток в разделе «Моя зарплата».

## 5. Роли

| Роль | Что видит |
|---|---|
| **Администратор** | всё |
| **Менеджер по оформлению** | заказы, обращения, футболки, задачи. Без зарплат, отчётов, сотрудников и настроек |
| **Исполнитель** | только фотопечать, задачи и свою зарплату |

Меню описано в одном файле `frontend/src/components/layout/navigation.ts` —
новый раздел добавляется одной строкой и сразу появляется везде: в боковом
меню, в мобильном и в нижней панели. Права меню — это только меню; сервер
проверяет доступ отдельно.

## 6. Сценарии оформления

Механизм, на котором будет построено рабочее окно менеджера.

Продукт описан **данными**, а не кодом: какие поля, какие вопросы задать
клиенту, когда поле появляется и когда оно обязательно. Описание отдаётся
браузеру по API, форма строится по нему.

```
scenarios/
  scenario.types.ts       описание сценария
  scenario.engine.ts      видимость, обязательность, «что осталось выяснить»
  scenario.registry.ts    реестр продуктов
  products/
    photo.scenario.ts     что спрашиваем по фотопечати
    photo.mapper.ts       как это становится заказом
    tshirt.scenario.ts
    tshirt.mapper.ts
```

**Добавить новый продукт** = два файла в `products/` и одна строка в реестре.
Фронтенд не трогаем.

Черновик оформления живёт на самой заявке (`scenarioKey`, `scenarioAnswers`),
а не в браузере: переписка с клиентом идёт часами, вкладку закрывают.
Оформление меняет статус той же записи — заявка становится заказом, а не
копируется в новую.

**Интерфейса под это пока нет.** Механизм работает только через API.

## 7. Внешние связи

### Партнёр по футболкам

Мы отправляем заказ в его систему (вебхук), он забирает ТЗ-фото и стикер по
нашему API. Ключ идемпотентности не даёт создать заказ дважды. Статусы
подтягиваем опросом раз в 30 секунд.

### Telegram

Рабочая группа получает: план дня, напоминания по задачам (каждый день, пока
задача не закрыта, с упоминанием ответственного) и напоминание попросить
отзыв у клиента — вместе с готовым текстом для отправки.

Прод режет доступ к Telegram, поэтому `api.telegram.org` прибит к рабочему
IP в `docker-compose.prod.yml`.

### Avito

Клиент API есть, ключи прописаны на сервере. Сейчас доступно бесплатное:
профиль и рейтинг с отзывами. **Переписка (мессенджер) требует платного
тарифа** — без него окно менеджера с чатом не запустить.

- `OrderPhoto` — заказ. Номер вида `ГГГГММ-NNN`, статус, суммы, сроки,

  таблицы оставлены с данными на случай возврата.

## 9. Развёртывание

```
рабочая машина → GitHub → сервер → Docker Compose
```

Продакшн: `195.2.75.249`, проект в `/opt/raspechatka`.

Порядок выкатки:

1. собрать локально (`npm run build` в `crm-new` и `frontend`);
2. залить `crm-new/dist`, `crm-new/src/generated`, `frontend/dist`;
3. `git pull` на сервере;
4. `docker compose -f docker-compose.prod.yml up -d --build`.

Миграции применяются сами при старте контейнера.

**Резервные копии:** `backup-db.sh` по расписанию в 03:00 каждый день,
складываются в `/opt/raspechatka/backups`.

## 10. Слабые места

Перечислено по важности. Это не список задач, а честная картина.

1. **Нет HTTPS.** Система работает по открытому HTTP. Пароли сотрудников и
   данные заказов идут по сети незашифрованными, их видно в любой публичной
   сети. Это же блокирует вебхуки Авито — они требуют защищённый адрес.
2. **Пароль администратора слабый** — это номер телефона. Подбирается.
3. **Резервные копии лежат на том же сервере.** Сервер потерян — потеряны и
   копии. Нужна выгрузка наружу.
4. **Всё на одной машине.** База, бэкенд и фронтенд в одном месте, запасного
   нет. Простой = остановка работы.
5. **Документация расползлась.** `SYSTEM_MAP.md`, `AGENTS.md`, два аудита,
   `REPORT.md` — часть устарела (например, там ещё живёт склад футболок).
   Стоит оставить один документ и вести только его.
6. **Мелкий беспорядок в миграциях**: две числятся применёнными, а папок нет;
   у двух разошлось содержимое. Работе не мешает — `migrate deploy` применяет
   только новое, — но при переносе на чистую базу вылезет.
