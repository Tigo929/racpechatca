# Интеграция с исполнителем-партнёром (печать футболок)

Документ описывает, как сторонняя система (исполнитель-партнёр, пункт печати и
самовывоза) подключается к CRM «Распечатка» для приёма заказов на футболки.
Написан так, чтобы разработчик или AI-агент на стороне партнёра мог реализовать
подключение без дополнительных вопросов.

## 1. Модель взаимодействия

Направление: **CRM «Распечатка» → партнёр**. Мы инициируем передачу.

1. Оператор в нашей CRM оформляет заказ на футболки, прикрепляет ТЗ-фото
   (согласованный с клиентом макет) и нажимает **«Отправить исполнителю»**.
2. Наш сервер делает **webhook POST** на эндпоинт партнёра
   (`PARTNER_WEBHOOK_URL`) с данными заказа и **ссылками** на ТЗ-фото и стикер.
3. Партнёр в ответ на webhook создаёт заказ у себя и по ссылкам **скачивает**
   ТЗ-фото и PDF-стикер с нашего API, авторизуясь своим токеном.

Файлы отдаются по ссылкам, а не в теле webhook (индустриальный стандарт:
webhook с URL-ами ресурсов) — так payload остаётся лёгким.

## 2. Аутентификация

Используются два независимых секрета:

| Секрет | Кто хранит | Где применяется |
|--------|-----------|-----------------|
| `PARTNER_WEBHOOK_TOKEN` | выдаёт партнёр, храним мы | мы шлём его в заголовке `X-Partner-Token` при POST на webhook партнёра — чтобы партнёр убедился, что запрос от нас |
| `PARTNER_API_TOKEN` | выдаём мы, хранит партнёр | партнёр шлёт его в заголовке `X-Partner-Token` (или `Authorization: Bearer <token>`) при загрузке заказа/файлов с нашего API |

Оба — статические строки, передаются вне канала (не в коде).

## 3. Webhook, который мы отправляем партнёру

```
POST {PARTNER_WEBHOOK_URL}
Content-Type: application/json
X-Partner-Token: {PARTNER_WEBHOOK_TOKEN}
```

Тело:

```json
{
  "event": "tshirt_order.ready",
  "external_request_id": "8f0e3d2c-1a2b-4c5d-9e6f-0a1b2c3d4e5f",
  "order": {
    "order_number": "20260718-005",
    "note": "Срочно, к пятнице",
    "tshirt_model": "Оверсайз 240",
    "items": [
      {
        "color": "Белый",
        "size": "M",
        "print_location": "FRONT",
        "print_location_label": "Грудь",
        "print_type": "DTF",
        "print_type_label": "DTF",
        "quantity": 3,
        "price_per_piece": 800,
        "line_total": 2400,
        "client_item": false
      }
    ],
    "stock": {
      "total_quantity": 5,
      "breakdown": [
        { "color": "Белый", "size": "M", "quantity": 3, "client_item": false },
        { "color": "Чёрный", "size": "L", "quantity": 2, "client_item": false }
      ]
    },
    "payment": {
      "order_total": 5000,
      "prepaid": 2500,
      "balance_due": 2500,
      "balance_on_pickup": true
    },
    "delivery": { "method": "PICKUP", "is_pickup": true },
    "files": {
      "tech_spec_url": "https://<наш-хост>/partner/orders/<id>/techspec-photo",
      "sticker_url": "https://<наш-хост>/partner/orders/<id>/sticker"
    }
  }
}
```

**Идемпотентность.** `external_request_id` стабилен между повторными отправками
одного заказа. При повторном получении того же `external_request_id` партнёр
**не должен создавать дубль** — обновите существующий заказ.

**Ожидаемый ответ.** Любой `2xx` = принято. Любой другой статус или таймаут
(15 с) мы фиксируем как ошибку и показываем оператору кнопку «Отправить
повторно». Тело ответа не обязательно, но приветствуется JSON вида
`{ "ok": true }`.

## 4. Эндпоинты, откуда партнёр забирает данные (pull)

Базовый адрес — наш публичный хост (`PUBLIC_BASE_URL`). Все запросы требуют
заголовок `X-Partner-Token: {PARTNER_API_TOKEN}`.

### 4.1 Данные заказа

```
GET {PUBLIC_BASE_URL}/partner/orders/{id}
```

Возвращает тот же объект `order`, что и в webhook (см. раздел 3). Полезно, если
партнёр хочет перезапросить заказ.

### 4.2 ТЗ-фото (макет)

```
GET {PUBLIC_BASE_URL}/partner/orders/{id}/techspec-photo
```

Отдаёт файл макета (`image/jpeg`, `image/png`, `image/webp` или
`application/pdf`). `404`, если фото не прикреплено.

### 4.3 Стикер (PDF, 58×40 мм)

```
GET {PUBLIC_BASE_URL}/partner/orders/{id}/sticker
```

Отдаёт `application/pdf` — термоэтикетку заказа (номер + штрихкод Code128,
позиции-футболки, итог, предоплата 50%, остаток при получении).

## 5. Справочник полей и перечислений

### items[] (позиции — ТОЛЬКО футболки)

| Поле | Тип | Описание |
|------|-----|----------|
| `color` | string | Цвет футболки (напр. «Белый») |
| `size` | string | Размер: `XS`,`S`,`M`,`L`,`XL`,`XXL`,`XXXL` |
| `print_location` | enum | Место печати (см. ниже) |
| `print_location_label` | string | То же по-русски (для удобства) |
| `print_type` | enum | Метод печати (см. ниже) |
| `print_type_label` | string | То же по-русски |
| `quantity` | int | Количество штук по позиции |
| `price_per_piece` | int | Цена за штуку, ₽ |
| `line_total` | int | `quantity × price_per_piece`, ₽ |
| `client_item` | bool | `true` — изделие клиента (его футболка); заготовку со склада НЕ списывать |

### stock (складская сводка для списания заготовок)

Готовая к учёту сводка — сколько и каких заготовок нужно для заказа, чтобы
партнёр списал их у себя на складе и вёл остатки.

| Поле | Тип | Описание |
|------|-----|----------|
| `total_quantity` | int | Общее количество футболок по заявке (все позиции) |
| `breakdown[]` | array | Разбивка по SKU: `color`, `size`, `quantity`, `client_item` |

`breakdown` агрегирован по паре **цвет + размер** (позиции с одинаковым
цветом и размером суммируются). Со склада партнёра списываются только строки
с `client_item = false`; при `client_item = true` заготовку приносит клиент.

`print_location`: `FRONT` (грудь), `BACK` (спина), `FRONT_BACK` (двусторонняя),
`SLEEVE_LEFT` (левый рукав), `SLEEVE_RIGHT` (правый рукав), `FULL` (полная
запечатка), `BY_TZ` (по ТЗ).

`print_type`: `DTF`, `DTG`, `SILK` (шелкография), `SUBLIMATION` (сублимация).

### payment

| Поле | Тип | Описание |
|------|-----|----------|
| `order_total` | int | **Полная** сумма заказа, ₽ (все позиции + доставка) |
| `prepaid` | int | Предоплата 50% (округление вверх), ₽ |
| `balance_due` | int | Остаток к оплате = `order_total − prepaid`, ₽ |
| `balance_on_pickup` | bool | `true` — остаток берётся при самовывозе |

> ⚠️ **Важно про суммы.** `order_total` — это ПОЛНЫЙ чек заказа. Партнёру
> передаются только футболочные позиции, поэтому сумма `line_total` по `items`
> может быть **меньше** `order_total` (в заказе могут быть иные, не
> производственные позиции). Для оплаты на самовывозе ориентируйтесь на
> `order_total` / `prepaid` / `balance_due`, а не на сумму позиций.

## 6. Переменные окружения (на нашей стороне)

| Переменная | Назначение |
|-----------|-----------|
| `PARTNER_WEBHOOK_URL` | эндпоинт партнёра, куда мы шлём заказы |
| `PARTNER_WEBHOOK_TOKEN` | токен, который партнёр ждёт от нас в `X-Partner-Token` |
| `PARTNER_API_TOKEN` | токен, которым партнёр авторизуется у нас |
| `PUBLIC_BASE_URL` | публичный адрес нашего API для ссылок на файлы |
| `UPLOAD_DIR` | каталог хранения ТЗ-фото на сервере |

Пока `PARTNER_WEBHOOK_URL` не задан, кнопка «Отправить исполнителю» пометит
заказ ошибкой с понятным текстом — данные не потеряются, отправку можно
повторить после настройки.
